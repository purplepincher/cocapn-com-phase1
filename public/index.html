cocapn.com · Phase 1

The boat as hermit crab.

cocapn is a Phase 1 sketch of a voice-first vessel: the captain speaks, the boat decides how to answer, and the same voice can move to a different hull later. Nothing below is built yet; every response is pre-written so you can see exactly how the machine would behave.

Why this is trustworthy

This is a deterministic simulation. Every word cocapn says, every button label, and every response you read was written by a human before you opened the page. No large language model is generating replies in real time. No physical boat exists. No captain has used this on the water. The honesty is not a caveat — it is the point. We are showing you the behavior of a designed machine so you can check it, not dressing up a plausible guess.

Two demos

Command-and-confirm loop — port ten, belay that

Command & Confirm Helm FSM

This demo is a five-state, thirteen-transition state machine running in your browser. The "realism" is in the timing and safety mechanics: a 500 millisecond trim pulse, a ten-second physical-override lockout, and a closed command grammar. It does not control anything. It shows how a deterministic helm might handle an order, a cancellation, a request for confirmation, and a human taking back the wheel.

Try it: say "port ten" and watch what happens →

Capability provisioning — deck lights, engine room, fishery

Deck-Lights / Engine-Room Provisioning

This demo is a fixed decision tree, not a conversation. Every branch, every cocapn line, and every artifact you see was authored in advance. You click labeled choices; the tree plays back the node that was already written. It demonstrates how a captain might provision a capability before handing the helm back to the command loop.

Try it: turn on the deck lights →

This is Phase 1: a sketch, hardened at purplepincher only if it proves out. The pattern this org already uses — build the smallest checkable thing here, then carry the real version forward — is exactly what you're looking at. Both demos are deterministic: the command loop is a closed-grammar state machine, the provisioning walkthrough is a fixed decision tree. Nothing is generated in response to what you click or type.
